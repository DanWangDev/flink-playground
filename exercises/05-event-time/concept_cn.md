# 练习 05 — 事件时间与水位线

## 你将学到

- **事件时间**与处理时间 — 为什么时间戳很重要
- **水位线** — Flink 如何跟踪事件时间的进度
- **侧输出** — 优雅地处理迟到数据

---

## 为什么这很重要

现实世界的数据很少完全有序。由于网络延迟、移动离线缓冲或上游背压，事件会延迟到达。事件时间处理确保无论到达顺序如何都能得到正确结果 — 这对金融计算、分析以及任何"发生时"比"收到时"更重要的用例至关重要。

---

## 核心概念

### 事件时间 vs 处理时间

| 时间 | 定义 | 用例 |
|------|------|------|
| 事件时间 | 嵌入在事件中的时间戳 | 精确分析、重放 |
| 处理时间 | 处理事件时的墙上时间 | 延迟敏感、近似 |
| 摄入时间 | 事件进入 Flink 的时间 | 两者之间的折衷 |

### 水位线

水位线是一个阈值，声明："不会再有早于 T 的事件到达。"

```
事件:     E1(1000)  E2(2000)  E4(4000)  E3(1500)  E5(5000)
水位线:    W(1999)      W(3999)                       W(4999)
窗口触发:              [0-3000)     [3000-6000)
```

E3 延迟到达（乱序），但在 2 秒的有界乱序容忍范围内，因此它仍然落在正确的窗口中。

### 侧输出

迟到事件（在水位线 + 允许的延迟之后到达）可以重定向到侧输出，而不是被静默丢弃：

```java
OutputTag<Event> lateTag = new OutputTag<Event>("late") {};
stream.keyBy(...)
      .window(...)
      .sideOutputLateData(lateTag)
      .process(...);

DataStream<Event> lateStream = result.getSideOutput(lateTag);
```

## 常见陷阱

1. **没有水位线 = 窗口永远不会关闭** — 没有水位线策略，事件时间窗口永远不会触发。始终分配时间戳和水位线。

2. **水位线是全局的** — 在多源作业中，水位线是所有分区中的最小值。一个慢分区会延迟整个作业。

3. **测试事件时间** — 使用固定时间戳和有界数据进行确定性测试。避免使用 `System.currentTimeMillis()`。

4. **水位线延迟** — 使用 `Duration.ofSeconds()` 设置水位线延迟，而不是 `Time.seconds()`。
